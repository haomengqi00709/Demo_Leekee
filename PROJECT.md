## 1. 背景

- **客户**：Winnie Xu（徐文怡），李记包装 / Leekee（上海东方国际 East Best · 广州李记包装）——**玻璃化妆品包装**供应商。
- **我方**：Trust AI Advisory（Jason Hao）。
- **来源**：本目录的 100 份 Excel + 一份通话录音 `Call with Winnie Xu.vtt`。

### 通话里确认的痛点与目标
- 现在做**一套报价要人工翻表约 1 小时**，容易出错；在卖的套系有 **600+ 套**，数据库很大。
- 大量重复的**兼容性问答**（这个泵能不能配这个瓶？这个盖能不能配这个泵？）靠人记，记不住。
- **目标**：做一个 agent，客户选定套系 + 配置后，**1 分钟自动出一份准确的报价单**，替代人工翻表。
- 定位：先做一个能演示的 **POC / demo**，拿给 Winnie 及其行业内朋友（下辖约 10 个销售）看，验证方向。

## 3. 编码规则

| 前缀 | 含义 |
|---|---|
| `H###` | **套系**（一整套产品组合，如 H563） |
| `CB####` / `CC####` | 瓶身 / 泵头 / 滴管 / 外罩 等（玻璃或注塑件） |
| `CA####` | 盖子 / 膏霜盖（含铝盖） |
| `SH####` | 部分单品（如滴管/泵组件） |
| 库存后缀 | 物料编码带批次后缀，如 `CC20695B` → `CC20695BHBAA` |

## 4. 报价计算逻辑（核心）

一个成品 = **瓶身 + 盖子 / 泵头 / 外罩 + 内塞 / 手拉垫**，价格层层叠加：

1. **光瓶价格**（bare bottle）。
2. **+ 蒙砂 / 门杀等基础工艺**（+0.15 或 +0.4）→ 得到**统一价格**。
   - 例：光瓶 2.4 + 门杀 0.4 = 统一报价 2.8。
3. **+ 盖 / 泵 / 外罩的加工工艺**：同一部件按工艺档位有一整列价格。
   - 例（H547 泵头）：全实色 0.95 → … → 泵身压嘴全电镀 1.85。
   - 例（H547 外罩）：全实色 1.6 → … → 透明幻彩电镀 3.0。
   - 工艺档位关键词：实色 / 电镀 / 过UV / 烫边 / 喷漆 / 手感漆(触感漆) / 哑光 / 幻彩镀 / 水转印(木纹)…
4. **+ 印刷**：普通印刷 ~0.2/次，烫金 ~0.25/次。
5. **× 数量档**：以 **5000 个**为基准价；2 万 / 5 万有阶梯折扣（单价递减）。
6. **× 客户加成**：不同客户 **+20% / +30% / +50%**。

**兼容性规则**：哪些泵/盖/外罩能配哪些瓶——目前是销售隐性知识，需沉淀为规则表。

## 5. 数据质量注意点（解析时的坑）

- **版本混乱**：有 `的副本`、`~$` 临时锁文件（忽略）；文件名日期跨 2022–2025；统一价年份混用（需归一到"最新有效价"）。
- **表是给人看的**：大量合并单元格、内嵌图片；**部件型号/容量只写在首行，后续工艺档位行靠向下继承**——解析要做 fill-down。
- **部分文件正文是图片**（如 `H582 Set Quotation`、wmf 图），文本抽不出，需 OCR 或人工补录。
- **币种混用**：对内 RMB，对外 USD/EUR，需汇率与 FOB 口径。
- **表头不统一**：不同文件列名/列序略有差异，需按表头关键词做列映射。

### 阶段 1 成果（2026-07-17）

数据库 `pricing_db/pricing.db`（SQLite），一次 `python3 build_db.py` 从 100 个 Excel 全量重建：

| 表 | 行数 | 内容 |
|---|---|---|

| `price_item_code` | 4,514 | 多码单元格("A/B")拆分 → 单编码查询 |
| `products` | 1,694 | 去重编码主表(类型/容量/供应商) |
| `glass_bottles` | 1,699 | 玻璃瓶单品(光瓶价+蒙砂=统一价) |

**视图**：`v_price_latest`(每个编码×工艺的最新有效价) · `v_series_components`(套系→部件) · `v_product_image`(编码→图)。

**查询工具**：`python3 query.py {stats|code CC20945|series H563|bottle 乳液瓶 100|search 滴泵}`。
